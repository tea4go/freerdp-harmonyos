## 根本原因（已确认）

**核心问题**: `freerdp_settings_get_uint32()` 函数内部使用 `WINPR_ASSERT` 进行参数验证，当检测到无效参数时触发 `abort()` 导致应用崩溃。

**错误做法**: 在回调函数中调用 `freerdp_settings_get_uint32()` 获取设置值：
```cpp
// ❌ 错误：会触发 abort()
UINT32 width = freerdp_settings_get_uint32(settings, FreeRDP_DesktopWidth);
```

**正确做法**: 参考 go-freerdp-webconnect 项目，直接访问 settings 结构体成员：
```cpp
// ✅ 正确：直接访问成员变量
UINT32 width = settings->DesktopWidth;
```

### 参考代码

通过分析 [go-freerdp-webconnect/rdp.go](C:/MyWork/GitCode/go-freerdp-webconnect/rdp.go)，发现：
1. ✅ Go 代码**从不调用** `freerdp_settings_get_*` 系列函数
2. ✅ 只在 `preConnect` 中使用 `freerdp_settings_set_*` 设置参数
3. ✅ 所有需要的值都是直接访问 settings 结构体成员

### 崩溃位置

在 HarmonyOS 代码中，有 **6 处**调用了 `freerdp_settings_get_uint32`：

1. ✅ `harmonyos_desktop_resize()` - 获取桌面尺寸
2. ✅ `harmonyos_post_connect()` - 获取桌面尺寸和色深
3. ✅ `harmonyos_allow_display_updates()` - 获取桌面尺寸（2处）
4. ✅ `harmonyos_resume_display_updates()` - 获取桌面尺寸
5. ✅ `harmonyos_request_full_screen_refresh()` - 获取桌面尺寸

## 技术分析

### 为什么 `freerdp_settings_get_uint32` 会触发 abort？

FreeRDP 的 `freerdp_settings_get_*` 系列函数内部使用 `WINPR_ASSERT` 宏：

```cpp
// FreeRDP 内部实现（简化）
UINT32 freerdp_settings_get_uint32(rdpSettings* settings, size_t id) {
    WINPR_ASSERT(settings != NULL);  // ← 如果 settings 无效，触发 abort()
    WINPR_ASSERT(id < FreeRDP_SETTINGS_MAX);
    // ... 其他验证逻辑
    return settings->values[id].u32;
}
```

在某些情况下（如 settings 对象部分初始化、线程安全问题、musl libc 兼容性等），这些断言会失败，导致 `abort()` 被调用。

### 为什么直接访问成员变量是安全的？

1. **结构体定义公开**: `DesktopWidth`、`DesktopHeight`、`ColorDepth` 等成员在 `settings_types_private.h` 中明确定义
2. **内存布局稳定**: 虽然标记为 `SETTINGS_DEPRECATED`，但仍然可用且稳定
3. **避免验证逻辑**: 直接访问绕过了可能触发 abort 的断言检查
4. **官方推荐**: go-freerdp-webconnect 等项目使用此方法，证明其可靠性
